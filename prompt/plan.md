# AI 实时会议翻译与总结助手方案计划书

## 1. 项目背景

在当前跨语言协作和远程会议场景中，会议内容存在以下问题：

- 多语言沟通成本高，中英混合会议较为常见。
- 人工记录会议纪要效率低且容易遗漏。
- 会后总结依赖个人整理，标准不统一。
- 关键信息无法实时捕捉，行动项容易滞后。

基于公司已有大模型能力与现有会议系统 Microsoft Teams，拟构建一套实时会议辅助系统。

## 2. 项目目标

构建一个“外挂式 AI 会议助手”，实现以下核心能力：

- 实时语音识别：会议音频转换为文本。
- 实时双语翻译字幕：支持中文与英文互译。
- 闭麦辅助表达：在 Teams 闭麦时，支持通过系统麦克风本地输入或键盘输入，实时生成英文说法。
- 自动会议总结：支持实时总结与会后总结。
- 自动提取 Action Items：识别任务项与待办事项。
- 自动生成结构化会议纪要。

## 3. 总体技术方案

### 3.1 系统架构

主流程：

```text
音频输入（系统 / 麦克风）
        ↓
实时语音识别（ASR）
        ↓
文本流处理（分句 / 缓存）
        ↓
┌───────────────────────────┐
↓                           ↓
实时翻译（公司 LLM API）     实时总结（公司 LLM API）
↓                           ↓
双语字幕输出                 会议摘要更新
└───────────────────────────┘
        ↓
UI 展示层（悬浮窗 / 面板）
```

补充输入链路（闭麦场景）：

说明：Teams 内的“闭麦”通常表示停止向会议发送麦克风音频，不等同于系统层面禁用麦克风。因此可以在本地继续使用系统麦克风做语音输入，或直接使用键盘输入，由助手生成适合会议场景的英文表达。

```text
输入方式 A：系统麦克风本地采集
用户说中文
        ↓
本地 ASR
        ↓
表达优化 / 即时翻译（公司 LLM API）
        ↓
输出可直接发送或朗读的英文表达

输入方式 B：键盘输入
用户键入中文
        ↓
表达优化 / 即时翻译（公司 LLM API）
        ↓
输出可直接发送或朗读的英文表达
```

## 4. 技术选型

### 4.1 音频采集

- Windows WASAPI loopback（系统音频）
- 虚拟声卡（VB-Cable）

目标：捕获 Microsoft Teams 会议音频。

### 4.2 语音识别（ASR）

方案：

- 主方案：Whisper / faster-whisper
- 可升级方案：流式 ASR（Azure Speech / 企业 ASR）

特点：

- 支持中英混合识别。
- 以 1 到 2 秒 chunk 进行处理。

### 4.3 翻译服务（核心依赖）

使用公司内部大模型 API。

实时翻译 Prompt 示例：

```text
你是实时会议翻译系统，请将以下内容翻译为英文：
要求：
- 保留技术术语
- 简洁自然
- 不添加解释

输入：
{text}
```

闭麦辅助表达 Prompt 示例：

```text
你是会议中的英文表达助手，请将用户输入的中文立即转换为适合会议交流的英文说法。
要求：
- 保持原意准确
- 表达自然、简洁、礼貌
- 优先使用口语化会议表达
- 不添加解释，不输出多个版本
- 如果输入是口语转写结果，自动修正明显口误或 ASR 噪声后再输出

输入：
{text}
```

### 4.4 会议总结模块

采用滑动窗口策略，窗口长度为 30 到 60 秒。

Prompt 示例：

```text
请对以下会议内容进行结构化总结：
1. 讨论主题
2. 核心结论
3. 待办事项（Action Items）

内容：
{text_window}
```

## 5. 功能设计

### 5.1 实时字幕界面

显示示例：

```text
【中文】
我们需要优化 IPC bus 的延迟

【英文】
We need to optimize IPC bus latency.
```

### 5.2 会议摘要面板

实时更新以下内容：

- 当前讨论主题
- 关键结论
- Action Items

### 5.3 会后输出

自动生成以下内容：

- Markdown 会议纪要
- TODO 列表
- 可导出 Jira / Confluence 的结构化内容

### 5.4 闭麦辅助表达

适用场景：

用户处于 Teams 闭麦状态，不希望语音直接进入会议，但仍希望通过本地系统麦克风说中文，或通过键盘输入中文，快速获得可直接用于英文会议沟通的表达。

交互方式：

- 模式 1：用户对着系统麦克风说中文，系统仅本地采集，不发送到 Teams。
- 模式 2：用户在输入框中键入中文。
- 系统将输入内容实时转换为适合会议语境的英文表达。
- 用户可将结果复制，或作为开麦前的表达参考。
- 支持短句场景，如澄清、确认、追问、延期、风险提示。
- 支持用户自定义不同的场景，例如：客户交流场景，供应商交流场景，国外技术专家，外部技术专家(用户可以更详细地描述场景和需求)

示例：

```text
【中文输入】
这个问题我们还需要再确认一下接口定义

【英文输出】
We still need to double-check the interface definition on this point.
```

价值：

- 降低中文母语用户在英文会议中的表达门槛。
- 减少临场组织英文措辞的时间。
- 与实时字幕能力互补，覆盖“听”和“说”两个环节。
- 在不打断会议的前提下，为闭麦状态提供独立的表达辅助通道。

## 6. 系统部署方式

### 方案 A（推荐）：本地客户端

- 技术栈：Python 或 Electron
- 本地音频采集
- API 调用公司模型

优点：

- 安全
- 无权限依赖
- 易部署

### 方案 B：企业插件版

- 集成到 Teams 或内部会议系统
- 作为“会议助手 Bot”运行

## 7. 性能指标

| 指标 | 目标 |
| --- | --- |
| 字幕延迟 | < 3 秒 |
| 翻译延迟 | < 1 秒 |
| 总体延迟 | 2 到 4 秒 |
| 识别准确率 | > 90%（技术会议） |

## 8. 风险分析

### 8.1 音频采集复杂

- Teams 音频获取依赖系统权限。
- 需虚拟声卡或 WASAPI 支持。
- 闭麦辅助表达若使用系统麦克风本地采集，需确认不会与 Teams 的设备占用或权限策略冲突。

### 8.2 实时性限制

- Whisper 并非严格流式。
- 需要进行 chunk 优化。

### 8.3 API 成本与频率

- 翻译与总结需做节流。
- 避免每个 token 都触发调用。

## 9. 可扩展方向

高级能力包括：

- 说话人识别（Who said what）
- 自动生成 Jira Ticket
- 技术关键词提取（IPC / AUTOSAR / RTOS）
- 多会议知识库沉淀
- 企业级会议搜索
