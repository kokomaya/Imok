# AI 实时会议翻译与总结助手方案计划书

## 1. 项目背景

在当前跨语言协作和远程会议场景中，会议内容存在以下问题：

- 多语言沟通成本高，中英混合会议较为常见。
- 人工记录会议纪要效率低且容易遗漏。
- 会后总结依赖个人整理，标准不统一。
- 关键信息无法实时捕捉，行动项容易滞后。
- 非英文母语者在英文会议中的实时表达存在门槛，影响沟通效率。

基于公司已有大模型能力与现有会议系统 Microsoft Teams，拟构建一套实时会议辅助系统。

## 2. 项目目标

构建一个"外挂式 AI 会议助手"，实现以下核心能力：

| 能力 | 说明 | 优先级 |
| --- | --- | --- |
| 实时语音识别 | 会议音频转换为文本 | P0 |
| 实时双语翻译字幕 | 支持中文与英文互译 | P0 |
| 闭麦辅助表达 | Teams 闭麦时，通过麦克风或键盘输入中文，实时生成英文表达 | P0 |
| 自动会议总结 | 支持实时滚动总结与会后完整总结 | P1 |
| 自动提取 Action Items | 识别任务项、责任人与截止时间 | P1 |
| 结构化会议纪要生成 | 自动输出可导出的会议纪要 | P2 |

## 3. 总体技术方案

### 3.1 系统架构

核心数据流：

```text
┌─────────────────────────────────────────────────────────┐
│                      音频采集层                          │
│  系统音频（WASAPI Loopback） ←── Teams 会议音频          │

└──────────────┬──────────────────────┬───────────────────┘
               ↓                      ↓
┌──────────────────────┐  ┌──────────────────────────────┐
│    VAD 语音活动检测    │  │    闭麦输入（语音/键盘）       │
│  （Silero-VAD）       │  │                              │
└──────────┬───────────┘  └──────────────┬───────────────┘
           ↓                              ↓
┌──────────────────────┐  ┌──────────────────────────────┐
│   ASR 语音识别        │  │   ASR / 直接文本输入          │
│  （faster-whisper）   │  │                              │
└──────────┬───────────┘  └──────────────┬───────────────┘
           ↓                              ↓
┌──────────────────────┐  ┌──────────────────────────────┐

└──────────┬───────────┘  └──────────────┬───────────────┘
           ↓                              ↓
   ┌───────┴───────┐            英文表达输出（可复制/TTS）
   ↓               ↓
 实时翻译        实时总结
（Streaming）   （增量合并）
   ↓               ↓
 双语字幕        会议摘要
   └───────┬───────┘
           ↓
┌──────────────────────────────────────────────────────────┐
```

### 3.2 闭麦辅助链路

说明：Teams 内的"闭麦"通常表示停止向会议发送麦克风音频，不等同于系统层面禁用麦克风。因此可以在本地继续使用系统麦克风做语音输入，或直接使用键盘输入，由助手生成适合会议场景的英文表达。

```text
输入方式 A：系统麦克风本地采集         输入方式 B：键盘输入
用户说中文                             用户键入中文
        ↓                                      ↓
本地 ASR（复用 faster-whisper）         直接文本输入
        ↓                                      ↓
        └──────────┬───────────────────────────┘
                   ↓
     表达优化 / 即时翻译（公司 LLM API, Streaming）
                   ↓
     输出英文表达 → 一键复制 / TTS 朗读（可选）
```

### 3.3 关键设计决策

| 决策点 | 选型 | 理由 |
| --- | --- | --- |
| 客户端框架 | Electron + Python 后端 | Electron 提供跨平台 UI，Python 生态成熟（音频处理/ASR） |
| 进程通信 | WebSocket（本地） | 前后端解耦，支持流式推送，延迟低 |
| 本地存储 | SQLite | 轻量、无需额外服务、支持全文检索 |
| LLM 调用方式 | Streaming SSE | 降低首 token 延迟，提升用户感知速度 |

## 4. 技术选型

### 4.1 音频采集

| 方案 | 用途 | 优先级 |
| --- | --- | --- |
| WASAPI Loopback | 捕获系统音频（Teams 会议声音） | 主方案 |
| PyAudio / sounddevice | 捕获本地麦克风（闭麦辅助输入） | 主方案 |
| 虚拟声卡（VB-Cable） | 系统音频采集的备选方案 | 降级方案 |

关键实现要点：

- 使用双通道采集：系统音频（会议内容）和麦克风音频（闭麦输入）独立采集，互不干扰。
- 音频格式统一为 16kHz、16bit、单声道 PCM，匹配 Whisper 输入要求。
- 采集缓冲区大小设置为 512 到 1024 帧，平衡延迟与稳定性。

### 4.2 语音识别（ASR）

**主方案：faster-whisper（本地推理）**

| 参数 | 推荐值 | 说明 |
| --- | --- | --- |
| 模型尺寸 | `large-v3` / `medium` | large-v3 精度最优；medium 适合无独显场景 |
| 计算后端 | CTranslate2 (CUDA / CPU) | 有 NVIDIA GPU 优先用 CUDA，否则 CPU int8 量化 |
| VAD 前端 | Silero-VAD | 过滤静音段，减少无效推理，降低延迟 |
| chunk 策略 | VAD 分句驱动 | 不再固定 1-2s 切分，改为由 VAD 检测语音段边界自动分句 |
| 语言设置 | `language=None`（自动检测） | 支持中英混合，无需预设语言 |
| beam_size | 3 | 平衡精度与速度 |

**VAD 分句策略（替代固定 chunk）：**

- Silero-VAD 检测语音起止点，以自然语句为单位送入 Whisper。
- 设置最大段时长 15s、最小静音间隔 300ms。
- 优势：避免将一句话切断导致识别错误，减少对静音段的无效推理。

**可升级方案：流式 ASR（Azure Speech / 企业 ASR）**

- 当需要更低延迟（<1s）或更高并发时，可切换为云端流式 ASR。
- 通过抽象 ASR 接口层实现无缝切换。

### 4.3 翻译服务（核心依赖）

使用公司内部大模型 API，**采用 Streaming SSE 方式调用**以降低首字延迟。

**术语表机制：**

维护一份可配置的领域术语表（JSON/YAML），注入 Prompt 中确保翻译一致性：

```json
{
  "IPC": "IPC",
  "AUTOSAR": "AUTOSAR",
  "总线": "bus",
  "中间件": "middleware",
  "功能安全": "functional safety",
  "软件架构": "software architecture"
}
```

实时翻译 Prompt：

```text
你是实时会议翻译系统。请将以下中文内容翻译为英文。

要求：
- 严格使用以下术语表中的译法：{glossary}
- 保留技术术语原文（如缩写）
- 简洁自然，符合技术会议语境
- 不添加解释或注释
- 如果输入内容过短或不完整，仍尽量翻译

上下文（最近 3 句已翻译内容，用于保持连贯性）：
{recent_context}

当前输入：
{text}
```

闭麦辅助表达 Prompt：

```text
你是会议中的英文表达助手。请将用户输入的中文立即转换为适合会议交流的英文说法。

当前场景：{scene_description}
（默认场景：跨国团队内部技术讨论会）

要求：
- 保持原意准确
- 表达自然、简洁、礼貌
- 优先使用口语化会议表达
- 不添加解释，不输出多个版本
- 如果输入是口语转写结果，自动修正明显口误或 ASR 噪声后再输出
- 严格使用以下术语表中的译法：{glossary}

输入：
{text}
```

**调用优化策略：**

- 翻译请求合并：相邻 500ms 内的短句合并为一次请求，减少 API 调用频率。
- 请求去重：与上一次翻译输入相同时跳过（ASR 可能重复输出同一段）。
- 超时降级：API 响应超过 3s 未返回时，显示原文并标记"翻译中"。

### 4.4 会议总结模块

**采用增量合并策略（替代简单滑动窗口）：**

```text
实时转写文本流
        ↓
按时间窗口分段（60s 一段）
        ↓
段落级摘要（每段生成 2-3 句要点）
        ↓
全局合并摘要（每 5 段触发一次全局合并，去重 + 结构化）
        ↓
最终输出：主题列表 / 关键结论 / Action Items
```

段落级摘要 Prompt：

```text
请从以下会议片段中提取关键信息：
- 讨论了什么主题
- 得出了什么结论（如有）
- 提到了什么待办事项或行动项（如有，标注责任人和时间）

会议片段：
{text_segment}
```

全局合并 Prompt：

```text
以下是会议各段的摘要，请合并为结构化的会议总结：

1. 讨论主题（合并相同主题，按重要性排序）
2. 核心结论
3. 待办事项（Action Items），格式：[ ] 事项描述 - @责任人 - 截止时间
4. 风险与待跟进项

各段摘要：
{segment_summaries}
```

## 5. 功能设计

### 5.1 实时字幕界面

**悬浮窗设计要点：**

- 半透明悬浮窗，始终置顶，可拖拽、可缩放。
- 最多显示最近 5 条字幕，自动滚动。
- 支持字体大小调节和透明度调节。
- 支持仅显示译文模式（减少视觉干扰）。

显示示例：

```text
[14:32:05] 张三
【中文】我们需要优化 IPC bus 的延迟
【英文】We need to optimize IPC bus latency.

[14:32:12] John
【英文】What's the current latency measurement?
【中文】当前的延迟测量结果是多少？
```

### 5.2 会议摘要面板

实时更新以下内容：

- 当前讨论主题（自动检测主题切换）
- 关键结论（按主题归类）
- Action Items（含责任人标注，如 ASR 可识别）
- 会议时间线（主题切换时间点）

### 5.3 会后输出

自动生成以下内容：

- Markdown 会议纪要（含时间戳、双语转写全文）
- 结构化 Action Items 列表（含责任人、截止时间）
- 可导出格式：Markdown / JSON / Confluence Wiki 格式
- 完整转写文本存档（SQLite 本地存储）

### 5.4 闭麦辅助表达

适用场景：

用户处于 Teams 闭麦状态，不希望语音直接进入会议，但仍希望通过本地系统麦克风说中文，或通过键盘输入中文，快速获得可直接用于英文会议沟通的表达。

交互方式：

- 模式 1：用户对着系统麦克风说中文，系统仅本地采集，不发送到 Teams。
- 模式 2：用户在输入框中键入中文。
- 系统将输入内容实时转换为适合会议语境的英文表达。
- 用户可将结果一键复制到剪贴板，或通过 TTS 朗读（可选）。
- 支持快捷键触发（如 `Ctrl+Shift+M` 激活闭麦输入面板）。
- 支持短句场景，如澄清、确认、追问、延期、风险提示。
- 支持用户自定义场景，例如：客户交流场景、供应商交流场景、国外技术专家、外部技术专家（用户可详细描述场景和需求，场景配置持久化保存）。
- 支持常用短语收藏与快捷输入。

示例：

```text
【中文输入】
这个问题我们还需要再确认一下接口定义

【英文输出】
We still need to double-check the interface definition on this point.

[📋 复制]  [🔊 朗读]
```

价值：

- 降低中文母语用户在英文会议中的表达门槛。
- 减少临场组织英文措辞的时间。
- 与实时字幕能力互补，覆盖"听"和"说"两个环节。
- 在不打断会议的前提下，为闭麦状态提供独立的表达辅助通道。

## 6. 系统部署方式

### 方案 A（推荐）：本地客户端（Electron + Python）

| 层级 | 技术 | 职责 |
| --- | --- | --- |
| UI 层 | Electron (HTML/CSS/JS) | 悬浮窗、摘要面板、配置界面 |
| 后端服务 | Python (FastAPI/WebSocket) | 音频采集、ASR、LLM 调用、数据处理 |
| 进程通信 | WebSocket (localhost) | 前后端实时双向通信 |
| 本地存储 | SQLite | 会议记录、配置、术语表 |

优点：

- 数据不出本地，安全性高。
- 无需 IT 审批额外权限。
- 支持离线 ASR（仅翻译/总结需联网）。
- 可独立于 Teams 版本更新运行。

打包分发：

- 使用 electron-builder 打包为 `.exe` 安装包。
- Python 后端通过 PyInstaller 打包或内嵌 Python 运行时。
- 提供一键安装脚本，自动检测 GPU 驱动并选择合适的 ASR 模型。

### 方案 B：企业插件版（远期）

- 集成到 Teams 或内部会议系统。
- 作为"会议助手 Bot"运行。
- 需要企业管理员权限审批。

## 7. 性能指标

| 指标 | 目标 | 测量方式 |
| --- | --- | --- |
| ASR 识别延迟 | < 2 秒（从说完到文本出现） | 端到端时间戳对比 |
| 翻译首字延迟 | < 1 秒（Streaming 首 token） | API 调用到首 token 返回 |
| 总体字幕延迟 | 2 到 4 秒（从说话到双语字幕出现） | 用户端实测 |
| ASR 识别准确率 | > 90%（技术会议场景） | 与人工转写对比 WER |
| 闭麦表达延迟 | < 2 秒（从输入到英文输出） | 端到端时间戳对比 |
| 内存占用 | < 2 GB（含 ASR 模型） | 系统监控 |
| CPU 占用 | < 30%（有 GPU）/ < 60%（纯 CPU） | 系统监控 |
| 单次会议时长支持 | ≥ 2 小时无中断 | 长时间稳定性测试 |

## 8. 风险分析与缓解策略

### 8.1 音频采集复杂

**风险：**

- Teams 音频获取依赖系统权限。
- WASAPI Loopback 可能受系统策略限制。
- 闭麦辅助使用系统麦克风，需确认不与 Teams 设备占用冲突。

**缓解：**

- 优先使用 WASAPI Loopback，失败时自动降级到虚拟声卡方案。
- 首次启动时执行音频采集自检，提示用户配置。
- 闭麦输入使用独立麦克风实例，与 Teams 不共享设备句柄。

### 8.2 实时性限制

**风险：**

- Whisper 为非流式模型，chunk 处理引入延迟。
- 无 GPU 环境下推理速度不足。

**缓解：**

- 引入 Silero-VAD 智能分句，减少无效推理（静音段跳过）。
- 无 GPU 时自动切换为 `medium` 或 `small` 模型 + int8 量化。
- 提供配置项允许用户在延迟与精度之间选择。

### 8.3 API 成本与频率

**风险：**

- 高频调用导致 API 配额耗尽或被限流。
- 网络抖动导致翻译中断。

**缓解：**

- 请求合并：500ms 内的短句合并为一次请求。
- 请求去重：相同文本不重复请求。
- 本地缓存：相同/相似翻译结果缓存复用。
- 限流保护：设置每分钟最大请求数（如 60 次/分钟），超出时排队处理。
- 断网降级：API 不可用时仅显示原文，恢复后自动重连。

### 8.4 数据安全与隐私

**风险：**

- 会议内容属企业敏感信息，经由 API 传输存在泄露风险。

**缓解：**

- ASR 完全本地执行，音频数据不出设备。
- LLM API 调用仅传输转写文本，不传输原始音频。
- API 调用走企业内网 / VPN，使用 HTTPS + API Key 认证。
- 本地存储的会议记录支持加密存储（可选）。
- 不记录、不上传原始音频文件。

## 10. 项目实施计划

### Phase 1a（核心链路验证，第 1 周）

**目标：** 跑通 "音频采集 → ASR → 文本输出" 核心链路。

- [ ] WASAPI Loopback 音频采集 + 格式转换（16kHz PCM）
- [ ] Silero-VAD 集成，实现语音段检测
- [ ] faster-whisper 集成，VAD 分句后送入识别
- [ ] 命令行输出实时转写文本（验证链路可用性）
- [ ] 基础错误处理（音频设备不可用提示）

**验收标准：** 能实时采集 Teams 会议音频并输出中英文转写文本，WER < 15%。

### Phase 1b（翻译 + 基础 UI，第 2 周）

**目标：** 完成实时翻译和闭麦辅助的最小可用版本。

- [ ] LLM 翻译服务对接（Streaming SSE）
- [ ] 翻译请求合并与去重
- [ ] 闭麦键盘输入 → 英文表达输出
- [ ] 闭麦麦克风输入 → ASR → 英文表达输出
